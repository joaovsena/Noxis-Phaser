ACHADOS IMPORTANTES DAS RODADAS
1. O servidor NAO parece ser o ponto principal da falha
- Logs mostram que handleCharacterEnter conclui.
- user_login sai normalmente.
- world_state meta foi gerado varias vezes com sucesso.

2. O cliente apresentava comportamento intermitente no recebimento da sequencia inicial
- Em algumas rodadas chegava auth_success e parava.
- Em outras, chegava world_static e parava.
- Em outras, o browser nem mostrava a mensagem seguinte ao auth_success.

3. Foi identificado um problema estrutural potencial
- O loop continuo do servidor enviava world_static/world_state enquanto o bootstrap de character_enter ainda estava em andamento.
- Isso criou suspeita forte de corrida entre:
  - bootstrap inicial
  - broadcast continuo do mundo

CORRECOES ESTRUTURAIS MAIS RECENTES
1. world_state.mobs foi sanitizado
- Objetivo: remover campos runtime brutos/problematicos.
- Resultado: bytes do world_state cairam bastante (~12.7 KB).
- Isso NAO resolveu sozinho.

2. bootstrap.initial unico foi tentado
- authSuccess + hotbarState + worldStatic + worldState + inventoryState num pacote so.
- Servidor enviou corretamente.
- Cliente continuou com falhas intermitentes para consumir.

3. bootstrap dividido em duas fases
- bootstrap.auth
  - authSuccess
  - hotbarState
- bootstrap.world
  - worldStatic
  - worldState
  - inventoryState

4. stream continuo do mundo agora deve ficar travado ate o cliente confirmar bootstrap.ready
- server/index.ts:
  - conexao recebe bootstrapReady = false
  - broadcast continuo ignora cliente enquanto bootstrapReady for false
- client/src/game/net/GameSocket.ts:
